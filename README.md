# redrob-ranker

Candidate ranking system for the India Runs Data & AI Challenge. Reads 100,000 candidate profiles, scores each against a Senior AI Engineer job description, and produces a 100-row `submission.csv` in under a minute on CPU.

No GPU needed. No external API calls during ranking. The heavy work (embedding 100K profiles) runs once offline.

---

## How it works

Two-phase design — offline precompute once, then rank fast:

**Phase 1 — precompute (run once, ~15–20 min):**
```
candidates.jsonl → precompute.py → artifacts/
                                    ├── embeddings.npy      # 100K × 384 float32
                                    ├── candidate_ids.npy
                                    ├── features.pkl        # structured scalar features
                                    └── jd_embedding.npy    # job description vector
```

**Phase 2 — rank (< 1 min every time after that):**
```
rank.py → loads artifacts → scores all 100K → top-100 → submission.csv
```

The scoring formula is:

```
final_score = (
    0.40 × semantic_similarity      # cosine(candidate_embedding, JD_embedding)
  + 0.30 × skill_score              # weighted required-skill match + duration depth
  + 0.30 × career_quality           # title fit, company type, experience band, education
) × behavioral_multiplier           # 0.3–1.2 based on 23 redrob_signals fields
  × honeypot_zero                   # 0 if flagged as trap profile, else 1
```

Candidates from the `DISQUALIFIER_TITLES` list who also claim 8+ AI skills are flagged as honeypots and zeroed out. Consulting-firm-only career paths get a 0.2× penalty. The behavioral multiplier accounts for notice period, response rate, activity recency, and interview completion.

## Quickstart

### Requirements

Python 3.10+ recommended. Install dependencies:

```bash
pip install -r requirements.txt
```

The first time `precompute.py` runs, it downloads `all-MiniLM-L6-v2` (~22 MB) and saves it to `artifacts/model/`. After that everything runs offline.

### Step 1 — precompute (do this once)

```bash
python precompute.py --candidates /path/to/candidates.jsonl
```

Go get a coffee. This takes 15–20 min depending on your CPU. Progress bar shows estimated time remaining. Output lands in `artifacts/`.

> If you cancel mid-way and re-run, it will redo everything from scratch. There's no checkpointing yet.

### Step 2 — rank

```bash
python rank.py --candidates /path/to/candidates.jsonl --out ./submission.csv
```

Typical runtime on a modern CPU: 30–60 seconds. The script runs `validate_submission.py` automatically at the end and will fail loudly if anything is wrong with the output format.

## Project structure

```
redrob-ranker/
├── rank.py                    # entry point — produces submission.csv
├── precompute.py              # run once offline — generates artifacts/
├── validate_submission.py     # copied from the challenge data folder
├── requirements.txt
├── submission_metadata.yaml   # fill this in before submitting
│
├── src/
│   ├── config.py              # all constants and weights — tune here, not in logic files
│   ├── data_loader.py         # streaming JSONL parser (never loads full 487MB into RAM)
│   ├── embedder.py            # sentence-transformer wrapper + candidate text builder
│   ├── features.py            # structured scalar features (experience, skills, career, location)
│   ├── behavioral.py          # redrob_signals → behavioral multiplier
│   ├── honeypot_detector.py   # rule-based trap detection
│   ├── scorer.py              # combines all signals into final score
│   ├── reasoner.py            # generates the reasoning column
│   └── utils.py               # date parsing, text normalisation, shared helpers
│
└── artifacts/                 # generated by precompute.py — not in git
    ├── embeddings.npy          # ~150 MB
    ├── candidate_ids.npy
    ├── features.pkl
    ├── jd_embedding.npy
    └── model/                  # sentence-transformer weights
```

## Tuning

All weights and keyword lists live in `src/config.py`. If the first run produces suspicious top-10 results (non-technical titles ranking high, wrong experience range, etc.), that's the file to change. Re-running `rank.py` picks up config changes immediately — no need to redo precomputation.

Things worth checking in the first output:
- Top-10 should all be in India (or `willing_to_relocate: true`)
- Experience range should be mostly 5–9 years
- None of the top-100 should have titles from `DISQUALIFIER_TITLES` unless they have a genuinely strong technical career history

If a consulting-firm candidate is ranking too high, lower `CONSULTING_ONLY_MULTIPLIER` in config. If notice period is being weighted too aggressively, adjust the notice band thresholds in `behavioral.py`.

## Submission output format

```
candidate_id,rank,score,reasoning
CAND_0001234,1,0.8841,"7.2-year ML Engineer at Razorpay; strong in FAISS, RAG, embeddings; Pune-based; recently active."
...
```

- 100 rows exactly (ranks 1–100)
- Scores non-increasing by rank
- Reasoning pulled from actual profile fields — no fabricated claims

Run the validator manually if you want to check before submitting:

```bash
python validate_submission.py submission.csv
```

## Notes

**Why `all-MiniLM-L6-v2`:** Fast enough to embed 100K candidates in ~14 seconds (post-precompute, during scoring). Small enough that the 22 MB weights download quickly on first run. Good enough that it handles paraphrasing well — a candidate who "built a recommendation engine for e-commerce" still gets a decent semantic match to "ranking systems for talent discovery."

**On the honeypot detection:** The dataset has ~80 planted trap profiles with impossible characteristics (expert-level skill with 0 months of use, overlapping full-time employment dates, non-technical titles with 12+ advanced AI skills). The detection runs as a hard pre-filter. Any candidate that triggers a honeypot rule gets score = 0 and cannot enter the top 100.

**RAM usage:** `precompute.py` streams candidates line-by-line and never loads the full file. Peak RAM during precompute is roughly 2–3 GB (mostly the embedding matrix being built up in memory). `rank.py` loads the pre-built `artifacts/` files, peak around 1.5 GB.

**If precompute crashes:** Most likely cause is a corrupted/incomplete `candidates.jsonl`. Malformed lines are skipped with a warning, but if a large block is corrupt the embeddings matrix dimensions won't match `candidate_ids.npy`. Delete `artifacts/` and re-run from scratch.
